## Rendering Rules

### Rich text placeholders

- While editing, text components should show raw placeholders such as `{name}`.
- On RTE exit/blur, the canvas should show resolved preview values.
- `templateText` is the authoritative raw template source and must survive preview rendering.
- Plain text datasource replacement must not run inside:
  - `.json-table-container`
  - `.json-table-wrapper`
  - `.json-data-table`
  - native table descendants
  - chart containers

### Datasource tables

- Live editor tables are generated by `customJsonTable`.
- Export must not trust stale live `innerHTML` when datasource-bound tables are involved.
- Export table rebuilds should use current JSON plus continuation/header subset state from `data-json-state`.
- For horizontally split wide tables, the rendered continuation table headers are authoritative because they identify the remaining column subset.
- For vertically paginated table fragments, export must infer the row start from serialized fragment data before rebuilding from bulk JSON, otherwise every fragment starts at row 0.
- Table styles can be CSS-rule based rather than inline; bulk export should copy computed cell styles before replacing table DOM.
- Meaningful datasource attributes that should remain on table nodes:
  - `my-input-json`
  - `data-json-file-index`
  - `data-json-state`
  - `data-source-table-id`

### Large-table performance

- The main UI freeze source is editor-side row/cell component creation, plus DataTables setup on large datasets.
- Current mitigation in active code:
  - large tables render a capped preview in the editor
  - full datasource state stays serialized for export
  - DataTables interactive setup is skipped for lightweight previews

### Page margins

- Page margins belong on `.page-content`, not `.page-container`.
- First-page margin bugs come from page-visual refresh paths that fail to reapply `.page-content` margin.
- Single-page PDF export can also lose margins if export CSS cleanup removes `margin` rules from `.page-content`; margin cleanup must only target outer `.page-container` wrappers.

### DOCX image export

- DOCX export is HTML-based through `html-docx-js`.
- Word support for inline SVG is unreliable.
- Any SVG used as:
  - inline `<svg>`
  - or `<img src="...svg...">`
  should be rasterized to PNG in the DOCX export preparation path.

## Bulk Export Architecture

### End-to-end flow

1. The client gathers a rendered snapshot from the live editor canvas.
2. `js/main.js` builds full export HTML including:
   - rendered page markup
   - editor CSS
   - export/runtime scripts
   - datasource bootstrap payload
3. For bulk export, the current mapped JSON object is injected into `window.__BULK_EXPORT_JSON__`.
4. In the export HTML runtime, datasource text and datasource tables are resolved client-side before final output is used.
5. For PDF generation, the client posts the final HTML to the PDF API as multipart form data.
6. The server returns a PDF blob.

### What is sent to the server

- A full HTML file, not partial fragments.
- Optional page payload metadata derived from `@page` sizing.

### What the server returns

- A PDF binary response on success.

### Where layout logic lives

- Page-break and table continuation logic is primarily client-side.
- The server mostly renders the already-prepared HTML.
- If preview/export layout diverges, inspect client-side HTML/runtime generation first.

## Known Correctness Notes

- Bulk export should prefer regenerated datasource DOM over cached live `innerHTML`.
- Continuation tables in export must honor rendered header keys first, then `data-json-state`, then raw JSON headers.
- Placeholder values should not resolve on click. Click/focus is edit mode. Blur/exit is preview mode.
- Object values are invalid for normal text placeholder replacement and must not become `[object Object]`.
- DOCX export should prefer rasterized image data for SVG sources.
